FEATURE 5 — Deterministic evaluation logic (filename-based + expected doc mismatch)
5) Determine per-item outcome from filename and expected type:
Given item.expected and uploaded file name (lowercased):
- Authenticity:
  - if filename includes "_fake" => authenticityFail = true
  - if filename includes "_real" => authenticityPass = true
  - else => authenticityUnknown (medium)
- Document mismatch:
  - expected tokens:
    - aadhaar: requires filename contains "aadhaar"
    - pan: contains "pan"
    - marriage: contains "marriage" OR "certificate"
    - birth: contains "birth" OR "certificate"
    - address: contains "address" OR "utility" OR "bill"
  - If expected is aadhaar and filename contains "voter" OR "pan" OR "dl" => mismatchFail
  - In general: if expected token not found but another reminder token found (e.g., voter/pan/dl) => mismatchFail with reason:
    “Expected Aadhaar, received Voter ID” etc.

Decision per item:
- If mismatchFail => FAIL (CRITICAL reason: “Document mismatch”)
- Else if authenticityFail => FAIL (CRITICAL reason: “Authenticity failed”)
- Else if authenticityUnknown => MEDIUM (eligible for MANUAL REVIEW)
- Else => PASS (LOW)

Compute riskScore per item:
- mismatchFail => 90–95
- authenticityFail => 85–92
- medium/unknown => 55–65
- pass => 10–20
